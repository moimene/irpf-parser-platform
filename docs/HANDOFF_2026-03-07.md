## Realidad de schema que condiciona el siguiente trabajo

El repo ya soporta un schema moderno, pero la produccion actual sigue operando sobre tablas legacy para acceso:

- legacy: `irpf_abogados`, `irpf_asignaciones`
- moderno local: `irpf_users`, `irpf_user_client_assignments`

La compatibilidad esta resuelta en:

- `apps/web/lib/access-store.ts`
- `apps/web/lib/client-store.ts`

Esto es funcional, pero es deuda tecnica. El siguiente trabajo de auth debe partir de esta realidad.

## Riesgos y deuda viva

1. Aun no existe una migracion aplicada en produccion que alinee completamente perfiles y asignaciones con el schema moderno.
2. No existe una vinculacion fuerte y persistida `auth.users.id -> perfil de despacho` como contrato definitivo.
3. El parser sigue limitado a `Pictet`, `Goldman Sachs` y `Citi`.
4. `714` y `720` ya tienen base de dominio canonica, pero el parser/UI todavia no escriben nativamente sobre ella.
5. No hay SSO corporativo ni gobierno avanzado de acceso.
